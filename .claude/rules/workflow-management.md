# 작업 관리 워크플로우

## 명령어
- `go`: plan.md에서 다음 미완료 테스트를 찾아 구현
- `status`: 현재 작업 현황 요약 출력
- `next`: 다음 작업 항목 확인

## 세션 시작 시
1. `.claude/plan.md` 파일을 읽어 현재 진행 상황 파악
2. 현재 Phase의 상세 계획 파일(`.claude/plan/phaseX-xxx.md`) 읽기
3. 체크리스트에서 다음 미완료 항목 찾기

## 작업 중
1. 각 작업 완료 시 `.claude/plan.md`의 체크리스트 업데이트
2. 중요한 결정이나 컨텍스트는 "컨텍스트 메모" 섹션에 기록
3. 블로커가 발생하면 "블로커 / 이슈" 섹션에 기록

## 세션 종료 전
1. 현재 진행 상황을 `.claude/plan.md`에 반영
2. 다음 작업 항목을 명확히 기록
3. 미해결 이슈가 있다면 기록

## Phase 완료 보고서 작성

### 목적
각 Phase 완료 시 보고서를 작성하여:
- 구현 내용을 문서화
- 사용자가 반드시 검토해야 할 코드를 식별
- 다음 Phase 개발자가 컨텍스트를 빠르게 파악

### 파일 구조
`.claude/reports/` 폴더에 두 파일 작성:
- `{phaseX}-{phase-name}-report.md` - 작업 보고서
- `{phaseX}-{phase-name}-diagram.md` - Mermaid 다이어그램

### 작업 보고서 (report.md)
**반드시 포함:**
- 구현된 기능 목록과 생성/수정된 파일
- **리뷰 필수 코드**: 보안, 성능, 정합성 측면에서 확인해야 할 코드를 `파일명:라인번호` 형식으로 명시
- 테스트 현황과 알려진 제한사항

**리뷰 필수 코드 판단 기준:**
- 보안: 사용자 입력 처리, 인증/인가, 민감 데이터 관련 코드
- 성능: 동시성 처리, 외부 시스템 호출, 반복문 내 I/O 코드
- 정합성: 트랜잭션 경계, 분산 시스템 데이터 일관성, 에러 복구 로직

### 다이어그램 (diagram.md)
**Mermaid 문법 사용** (GitHub, Notion 렌더링 지원)

**반드시 포함:**
- 컴포넌트 관계도: 클래스/인터페이스 의존성
- 시퀀스 다이어그램: 주요 유스케이스별 호출 흐름

### 하지 말 것
- 모든 코드를 나열하지 말고 핵심과 리뷰 필요 코드만 포함
- 다이어그램을 ASCII 아트로 작성하지 말고 Mermaid 사용
- 테스트가 없는 상태로 보고서 작성하지 않음
